ふだんの自分は、アプリ寄りのエンジニア研修講師だ。

コードも書くけれど、ここ数年で一番長く触っているのはエディタではなくスライドとカリキュラムの方。
設計や実装の話を、人に伝わる単位までほぐして並べ直す仕事をしている。

そんな自分に、ある日「インフラ講師できます？」という相談をいただいた。

オンプレの基盤設計やクラウドサーバーの運用経験もあるし、AWS の指導をしたこともあるし、コミュニティにも関わっているので、まったくの畑違いではない。
とはいえ、肩書としてはずっと「アプリ講師」を名乗ってきた側だ。

そこでインフラ研修の設計と講義を任されることになった。
この記事は、そのときのメモと、そこから見えた「アプリとインフラの距離感」の話になる。

⸻

アプリ講師がインフラを教える、という立ち位置

フリーランスとして経歴を説明するとき、アプリ講師としての話はしやすい。
• どんな技術スタックを扱ってきたか
• どういうレベルの受講生に向けて教えてきたか
• どのあたりのフェーズを得意としているか（基礎／応用／現場寄せ など）

このあたりは、案件を重ねるほど整理されていく。

一方で、インフラ側もまったくのゼロというわけではない。
AWS の講座を担当したり、ピンポイントでクラウド寄りの話をする機会はあった。

とはいえ、アプリ研修と並べてみると、インフラ研修の案件数は明らかに少ない。
「職種としての看板」を問われたときに、自然とアプリ講師側で名乗ってきた、という感じに近い。

そこにインフラ研修の話が来たので、
• 自分の守備範囲をもう一段整理し直せる
• アプリ寄りだけれどインフラも教えられる、と実績ベースで語れる

という意味で、最初の感情はかなりポジティブだった。

⸻

ハンズオンほぼなしという設計上の制約

一方で、不安要素もあった。

今回の研修はさまざまな事情が重なり、いわゆる「じっくりハンズオン」は難しい前提だった。
受講生が自分の環境でコマンドを叩きながら学ぶ、という形式にはしづらい。

インフラに限らず、アプリでもそうだが、手を動かせない学習には不安がつきまとう。
• 座学だけで本当に定着するのか
• 「分かった気がする」で終わらないか
• 実際に自分の手で試したときに、どこまで耐えられるのか

正直、「これで実力なんてつくわけないよな」という気持ちはかなり強かった。

とはいえ、外側の条件は簡単には変えられない。
そこで今回は、割り切って「座学＋テスト設計」に振り切ることにした。
• インフラの基本概念を、アプリの文脈とも行き来しながら説明する
• コマンドや画面は要点だけデモで見せる
• 最後に、理解していないと解けないテストを用意して確認する

「ハンズオンの不足分を、テスト設計と説明の精度でどこまで埋められるか」というチャレンジに寄せたかたちだ。

⸻

テスト結果が教えてくれたこと

研修当日は、スライドとホワイトボードとデモで進行した。
構成だけ見れば、アプリ研修とそこまで変わらない。

違うのは、扱う題材がインフラ寄りであることと、演習の比率がかなり低いこと。

最後に、事前に用意していたテストを受けてもらった。
単なる用語チェックではなく、
• 概念を正しく結びつけられるか
• 前提を押さえたうえで判断できるか

を確認する問題を混ぜた。

集計していると、ひとつだけ満点の答案があった。

人数規模のある研修で、内容を盛り込んだテストを満点で取り切る。
当然、受講生本人の努力あっての結果ではあるが、講師側から見ても「ここまで通るのか」と思える出来だった。

この瞬間に、
• ハンズオン少なめの研修でも、設計しだいで理解はここまで届く
• アプリ寄りの講師がインフラを教えても、きちんと成果を出せる

という感触を得られた。

「ハンズオンがないから無理」と言い切らず、条件の中で設計を組み替えたのは良かった、と思えたポイントでもある。

⸻

講師がいちばん学ぶ時間

インフラの内容を教えるためには、自分の理解を一度ばらして並べ替える必要がある。
• どの順番で話すと、前提の抜けが少ないか
• アプリ寄りの人にとって、どこが「暗黙のブラックボックス」になりがちか
• どこまで深掘りすれば今回の目的に対して十分か

このあたりを考えている時点で、けっこうな学びになる。

実際の講義中も、受講生から出る質問に答えながら、

「あ、自分はここを前提として話していたな」

と気付く場面が何度もあった。

講師なのに自分がいちばん学んでいる、という感覚はよくある。
インフラ寄りの内容だと、その度合いが少し強かっただけかもしれない。

インフラコミュニティや AWS 指導で触れてきた知識を、「受講生に渡せる単位」に再構成し直す作業は、なかなかに良い復習になる。

⸻

アプリとインフラの前提の違い

今回の研修で改めて整理したのは、「変化」に対する前提の違いだった。

アプリ側の設計は、ざっくり言うと「変化に強くある」ことを前提にしている。
• 仕様変更
• 機能追加
• UI の調整
• A/B テスト

こういった変化が日常的に発生する前提で、リリースサイクルやテスト戦略を組み立てる。

一方で、インフラ側は「変化を制御する」ことが前提になる。
• 勝手に変わらないこと
• 想定外の挙動が出ないこと
• どこを変えればどこに影響が出るかを把握しておくこと

このあたりを崩さないように、変更の入り方をコントロールする世界だ。

どちらが正しい／強いという話ではなく、前提にしている「変化の扱い」が違う。

今回の研修は、この違いを自分なりの言葉に落とし直す作業でもあった。

⸻

アプリ寄りのエンジニアへのメモ

アプリ寄りのエンジニアから見ると、インフラは
• なんとなく怖い
• 壊したらいけないものが多そう
• 下手に触るくらいなら専門の人に任せたい

という存在になりがちだと思う。

その感覚は自然だし、自分も最初はそうだった。

今回インフラ研修を設計してみて感じたのは、
「全部触れるようになる必要はないけれど、隣の世界の前提くらいは共有しておくと楽になる」ということだ。
• 何を「壊したくない」と思っているのか
• どこまでがアプリ側の裁量で、どこからがインフラ側の領域か
• どんな変化が一番怖いのか

このあたりを少し知っておくだけでも、アプリ側の設計や相談の仕方が変わる。

インフラ研修は、受講生にとってその入口になるし、講師にとっても自分の理解を再構成し直すいいきっかけになる。

アプリ講師を自認している自分がインフラ研修を担当してみて、一言で得たものをまとめるなら、

「隣のレイヤーの前提を、実体験ベースで語れるようになった」

くらいだと思う。

インフラが気になっているアプリエンジニアも、インフラはちょっと怖いと感じている人も、一度どこかの研修や書籍で「前提の世界」を覗いてみるのは悪くない。
そのときたぶん、誰かより先に自分がいちばん学ぶことになるので、講師とは教える仕事ではなく誰よりも学ぶ仕事だと考えている。
